# Проектная работа: Redux-хранилище Stellar Burger

Вы подошли к завершающему проекту модуля. Stellar Burgers — самая настоящая космическая бургерная: где-то на краю Вселенной любая форма жизни может заказать тот бургер, который она хочет. Конечно, с помощью приложения.

В ходе проектной работы вы:

- подключите роутер и реализуете маршруты;
- добавите глобальное состояние и подключите к страницам;
- реализуете механизм авторизации и защитите личный кабинет пользователя.

> Так как оставшийся для реализации код достаточно сильно связан между собой, продвигайтесь с двух сторон.
>
> Сначала добавьте все маршруты и создайте компоненты страниц для них, но пока без реализации. Вторым шагом создайте слайсы, подключите к ним данные из API. Затем подключите данные к страницам и уже после этого выводите с использованием компонентов из UI.

## Что нужно сделать

В первую очередь ознакомьтесь с макетом.

Вам нужно реализовать навигацию и отображение данных с сервера на страницах конструктора, ингредиента, ленты заказов и личного кабинета.

## Функциональные требования

- Редактирование данных пользователя работает корректно:
  - На странице профиля пользователя в поля формы подставляются данные пользователя. Если пользователь отредактировал данные в форме, появляются кнопки «Отмена» и «Сохранить».
  - При нажатии кнопки «Сохранить» на странице профиля пользователя отправляется соответствующий запрос к API, который содержит изменённые данные.
  - При нажатии кнопки «Отмена» значения полей формы возвращаются в состояние до редактирования, а кнопки «Отмена» и «Сохранить» скрываются.
  - В шапке панели навигации активна та ссылка на маршрут, где сейчас находится пользователь. Ссылка остаётся активной, если пользователь попадает на вложенные динамические маршруты.
  - При клике на пункт «История заказов» происходит переход на экран `/profile/orders`, ссылка «История заказов» становится активной, экран `/profile/orders` пока отображается пустым.
  - При клике по ингредиенту происходит переход на маршрут `/ingredients/:id` и открывается модальное окно с описанием ингредиента. При прямом переходе на этот маршрут открывается страница с информацией об ингредиенте.
  - Заказы могут делать только авторизованные пользователи, все остальные действия доступны и неавторизованным.
  - События отправки форм отслеживаются по `onSubmit` формы.
- Экран «Лента заказов» работает корректно:
  - Отображается по маршруту `/feed`, доступен всем пользователям.
  - Обновляется в режиме реального времени, когда авторизованный пользователь создаёт заказ. Содержит подсчёт стоимости каждого заказа, исходя из стоимости его ингредиентов.
  - Отображает количество всех заказов, а также количество заказов за день.
- Экран «История заказов» работает корректно:
  - Отображается по маршруту `/profile/orders`, доступен только авторизованным пользователям.
  - Обновляется в режиме реального времени и отображает статусы заказов.
  - Содержит подсчёт стоимости каждого заказа, исходя из стоимости его ингредиентов. Статусы «Отменён», «Готовится» и «Выполнен» формируются в зависимости от значения поля `status` каждого заказа.
- При клике по заказу открывается модальное окно с информацией о заказе и происходит переход на маршрут `/feed/:number` или `/profile/orders/:number` — в зависимости от того, на какой странице пользователь кликнул по заказу. При прямом переходе на эти маршруты открывается страница заказа.
- Маршрут `/profile/orders/:number` доступен только авторизованным пользователям.

## Запросы к серверу

Пора переходить к запросам. Сами запросы к серверу уже прописаны в `utils/burger-api.ts`, вам необходимо использовать их вместе с Redux-функционалом.

Создайте `RootReducer` и подключите его к store. Далее создайте слайс, используя `createSlice`, и опишите в нём конкретные редюсеры для сохранения данных в store. Также в файле слайса создайте асинхронную Thunk-функцию, которая будет использовать нужный метод API и запрашивать данные с сервера. Непосредственно в самом компоненте используйте эту Thunk-функцию для получения самих данных.

Не забудьте, что теперь данные получаются и сохраняются в store, поэтому, чтобы отобразить их на экране, используйте селекторы.

Обратите внимание, что для хорошего UX стоило бы использовать лоадеры в те моменты, когда мы отправили запрос и ждём ответ от сервера. Поэтому не забудьте в экшен и редюсер добавить нужный функционал.

## Шаг 2. Настройка запроса авторизации

Если пользователь не залогинен, ему доступен основной функционал страницы:

- он видит конструктор и ингредиенты на главной странице;
- может собирать бургер;
- видит ленту заказов.

Остальные роуты должны быть закрыты авторизацией. В частности при нажатии на кнопку «Оформить заказ» или при попытке перейти в личный кабинет. Если пользователь не залогинен, его нужно перенаправлять на форму авторизации.

Это связано с тем, что нам нужно обязательно знать конкретного пользователя, который запрашивает данные, чтобы отдать ему именно его данные или историю заказов.

Ещё это вопрос безопасности. Мы должны точно знать, что перед нами именно этот пользователь, чтобы не отдать приватные данные другому лицу.
